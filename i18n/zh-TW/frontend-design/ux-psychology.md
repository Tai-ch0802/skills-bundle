# UX 心理學參考指南

> 深入探討 UX 定律、情感設計、信任建立以及行為心理學。

---

## 1. 核心 UX 定律

### 希克定律 (Hick's Law)

**原則：** 做決策所需的時間隨著選擇數量的增加而呈對數增長。

```
決策時間 = a + b × log₂(n + 1)
其中 n = 選擇的數量
```

**應用：**
- 導航：頂層項目最多 5-7 個
- 表單：拆分為多個步驟（漸進式揭露）
- 選項：儘可能提供預設選擇
- 篩選器：優先顯示最常用的，隱藏進階選項

**範例：**
```
❌ 錯誤：單個導航列中有 15 個選單項目
✅ 正確：5 個主要類別 + 「更多」

❌ 錯誤：一次顯示 20 個表單欄位
✅ 正確：3 步嚮導，每步 5-7 個欄位
```

---

### 費茲定律 (Fitts' Law)

**原則：** 到達目標所需的時間是目標距離與大小的函數。

```
MT = a + b × log₂(1 + D/W)
其中 D = 距離， W = 寬度
```

**應用：**
- 號召性用語 (CTA)：讓主要按鈕更大（最小高度 44px）
- 觸控目標：行動裝置上最小 44×44px
- 放置位置：重要動作應靠近游標自然位置
- 角落：「魔力角」（無限邊緣 = 容易點擊）

**按鈕尺寸：**
```css
/* 根據重要性調整大小 */
.btn-primary { height: 48px; padding: 0 24px; }
.btn-secondary { height: 40px; padding: 0 16px; }
.btn-tertiary { height: 36px; padding: 0 12px; }

/* 行動裝置觸控目標 */
@media (hover: none) {
  .btn { min-height: 44px; min-width: 44px; }
}
```

### 米勒定律 (Miller's Law)

**原則：** 一般人只能在工作記憶中保留 7±2 個資訊塊。

**應用：**
- 列表：分組成 5-7 個項目的區塊
- 導航：選單項目最多 7 個
- 內容：使用標題拆分長內容
- 電話號碼：555-123-4567（區塊化）

**區塊化範例：**
```
❌ 5551234567
✅ 555-123-4567

❌ 無間斷的長段文字
✅ 短段落
   搭配清單符號
   以及次標題
```

---

### 馮·雷斯托夫效應 (Von Restorff Effect / 隔離效應)

**原則：** 突出的項目更有可能被記住。

**應用：**
- CTA 按鈕：使用與其他元素截然不同的顏色
- 價格：突出顯示推薦方案
- 重要資訊：視覺差異化
- 新功能：徽章 (Badge) 或標註

**範例：**
```css
/* 所有按鈕為灰色，主要按鈕突出 */
.btn { background: #E5E7EB; }
.btn-primary { background: #3B82F6; }

/* 突出顯示推薦方案 */
.pricing-card { border: 1px solid #E5E7EB; }
.pricing-card.popular { 
  border: 2px solid #3B82F6;
  box-shadow: var(--shadow-lg);
}
```

---

### 序列位置效應 (Serial Position Effect)

**原則：** 列表開頭（首因效應）和結尾（近因效應）的項目最容易被記住。

**應用：**
- 導航：最重要的項目放在第一和最後
- 列表：關鍵資訊放在頂部和底部
- 表單：最關鍵的欄位放在開頭
- CTA：在長頁面的頂部和底部重複

**範例：**
```
導航：首頁 | [關鍵項目] | 聯絡我們

長著陸頁：
- Hero 區域的 CTA (頂部)
- 內容區塊
- 底部重複 CTA
```

---

### 雅各布定律 (Jakob’s Law)

**原則：** 使用者大部分時間都花在其他網站上。他們希望你的網站運作方式與他們已知的所有其他網站相同。

**應用：**
- **模式：** 對於搜尋列和購物車，使用標準的放置位置。
- **心理模型：** 利用熟悉的圖示（例如：放大鏡）。
- **詞彙：** 使用「登入」而不是「進入門戶」。
- **佈局：** 將標誌放在左上角以便返回「首頁」。
- **互動：** 向右滑動以返回/前進應具有原生感。
- **回饋：** 標準顏色（紅色 = 錯誤，綠色 = 成功）。

**範例：**
```
❌ 錯誤：點擊標誌後會帶你進入「關於我們」頁面的網站。
✅ 正確：點擊標誌始終讓使用者返回首頁。

❌ 錯誤：使用「星星」圖示代表「刪除」。
✅ 正確：使用「垃圾桶」圖示代表「刪除」。
```

---

### 泰斯勒定律 (Tesler’s Law / 複雜性守恆定律)

**原則：** 對於任何系統，都存在一定數量的複雜性是無法減少的，只能從使用者轉移到軟體。

**應用：**
- **後端：** 讓系統處理格式（例如：貨幣）。
- **偵測：** 通過郵遞區號自動偵測信用卡類型或城市。
- **自動化：** 預填回頭客資料。
- **個人化：** 根據之前的回答僅顯示相關欄位。
- **預設值：** 為常用設定提供智慧預設值。
- **整合：** 使用 SSO (社群登入) 來減輕註冊摩擦。

**範例：**
```
❌ 錯誤：要求使用者在表單的每個價格欄位前輸入 "USD $"。
✅ 正確：應用程式根據使用者位置自動加上 "$" 前綴。

❌ 錯誤：強迫使用者手動選擇他們的「卡片類型」（Visa/Mastercard）。
✅ 正確：根據輸入的前四位數字自動偵測卡片類型。
```

---

### 帕金森定律 (Parkinson’s Law)

**原則：** 工作會一直擴展，直到用完所有可用的時間。

**應用：**
- **效率：** 使用「自動儲存」來減少任務完成時間。
- **速度：** 限制轉換漏斗中的步驟。
- **清晰度：** 使用清晰的標籤，防止使用者為了理解含義而「游標試探」。
- **回饋：** 即時驗證，防止使用者在錯誤上浪費時間。
- **引導：** 為進階使用者提供快速的「特快」設置。
- **約束：** 在輸入框設置字數限制以便集中思緒。

**範例：**
```
❌ 錯誤：一個長達 10 頁的註冊表單，允許使用者瀏覽離開並遺失資料。
✅ 正確：使用 Google 或 Apple ID 的「一鍵登入」。

❌ 錯誤：給使用者無限的時間來填寫自傳。
✅ 正確：提供「建議自傳」功能幫助他們在幾秒鐘內完成。
```

---

### 多爾蒂門檻 (Doherty Threshold)

**原則：** 當電腦及其使用者以確保兩者都不必等待對方的步調（<400ms）進行互動時，生產力會大幅飆升。

**應用：**
- **回饋：** 為點擊提供即時的視覺提示。
- **載入：** 使用骨架屏 (Skeleton screen) 提高感知效能。
- **樂觀 UI：** 在伺服器回應前更新 UI（Optimistic UI）。
- **動效：** 使用微互動動效來掩蓋輕微的延遲。
- **快取：** 在背景預載下一頁或資源。
- **優先順序：** 在重型高解析度圖片之前載入文字內容。

**範例：**
```
❌ 錯誤：按鈕在點擊後 2 秒內沒有任何反應。
✅ 正確：按鈕立即變色並顯示「載入中」旋轉圖示。

❌ 錯誤：抓取資料時出現純白螢幕。
✅ 正確：骨架屏顯示內容將出現位置的灰色輪廓。
```

---

### 波斯塔爾定律 (Postel’s Law / 穩健性原則)

**原則：** 輸出時要保守，輸入時要開放（寬容地接受他人的輸入）。

**應用：**
- **錯誤處理：** 不要因為少了一個空格或橫線就報錯。
- **格式：** 接受 DD/MM/YYYY 或 MM/DD/YYYY 的日期。
- **輸入：** 自動刪除前後空格。
- **備案：** 如果使用者未上傳圖片，使用預設大頭貼。
- **搜尋：** 接受拼字錯誤並提供「你的意思是...？」建議。
- **無障礙：** 確保網站能在所有瀏覽器和裝置上運作。

**範例：**
```
❌ 錯誤：因為使用者在電話號碼中加了空格而拒絕。
✅ 正確：接受輸入並自動刪除空格。

❌ 錯誤：強迫使用者輸入 "January" 而不是 "01" 或 "Jan"。
✅ 正確：日期欄位能理解這三種格式。
```

---

### 歐卡姆剃刀 (Occam’s Razor)

**原則：** 在預測效果同樣良好的競爭假設中，應選擇假設最少的那個。最簡單的解決方案通常是最好的。

**應用：**
- **邏輯：** 移除不必要的點擊。
- **視覺：** 僅使用嚴格必要的字體/顏色。
- **功能：** 如果一個欄位能完成兩個欄位的工作，就合併它們。
- **文案：** 使用儘可能短的文字來傳達含義。
- **佈局：** 移除不服務於目標的裝飾性元素。
- **流程：** 除非絕對必要，否則避免分支路徑。

**範例：**
```
❌ 錯誤：一個「登入」按鈕會開啟新頁面，先輸入信箱，再輸入密碼。
✅ 正確：單個登入彈窗，在一個畫面中詢問兩者。

❌ 錯誤：在單張卡片上使用 5 種不同字體大小和 4 種顏色。
✅ 正確：使用 2 種字體大小和 1 種強調色。
```

## 2. 視覺感知 (格式塔原則 / Gestalt Principles)

### 接近原則 (Law of Proximity)

**原則：** 彼此靠近或鄰近的物件往往會被看作一組。

**應用：**
- **分組：** 保持標籤物理上靠近輸入欄位。
- **間距：** 在不相關的內容塊之間使用較大的邊距。
- **卡片：** 卡片內的文字應比邊框更靠近其圖片。
- **頁尾：** 將法律連結聚集在一起，遠離社群連結。
- **導航：** 將「使用者」設定與「應用程式」設定分開分組。
- **表單：** 將地址欄位分組在一起，與信用卡欄位分開。

**範例：**
```
❌ 錯誤：表單中每行文字之間都有巨大的、相等的間隙。
✅ 正確：標籤與其輸入框之間間距緊密，而成對項目之間間距較大。

❌ 錯誤：一個「提交」按鈕漂浮在頁面中央，遠離表單。
✅ 正確：「提交」按鈕直接放在最後一個輸入欄位下方。
```

---

### 相似原則 (Law of Similarity)

**原則：** 人眼往往將設計中相似的元素感知為一個完整的圖片、形狀或組塊，即使這些元素是分開的。

**應用：**
- **一致性：** 所有可點擊連結使用一致的顏色。
- **圖示：** 一組圖示應具有相同的線條粗細。
- **按鈕：** 具有相同重要性的按鈕使用相同的形狀/尺寸。
- **排版：** 所有章節標題使用相同的 H2 樣式。
- **回饋：** 所有「刪除」動作應使用相同的顏色（例如：紅色）。
- **狀態：** 懸停 (Hover) 和點擊 (Active) 狀態在整個 App 中必須一致。

**範例：**
```
❌ 錯誤：有些連結是藍色的，有些是綠色的，有些只是粗體黑色。
✅ 正確：App 中每個可點擊的文字元素都使用相同的藍色調。

❌ 錯誤：將「藍色按鈕」用於「提交」，同時也用於「取消」。
✅ 正確：「提交」是實心藍色；「取消」是藍色外框（幽靈按鈕）。
```

---

### 共同區域原則 (Law of Common Region)

**原則：** 如果元素共享一個具有明確定義邊界的區域，則往往會被感知為一組。

**應用：**
- **容器化：** 使用卡片將圖片和標題分組。
- **邊框：** 使用線條將側邊欄與主動態饋送分開。
- **背景：** 為頁尾使用不同的背景顏色。
- **彈窗：** 使用獨特的框將彈窗與頁面分開。
- **列表：** 為行使用交替的背景顏色（斑馬條紋）。
- **標頭：** 在頂部放置一條實心條狀區域以分組導航項目。

**範例：**
```
❌ 錯誤：新聞文章列表中，不同故事的文字和圖片重疊。
✅ 正確：每篇文章都包含在淺灰色背景上的白色卡片中。

❌ 錯誤：頁尾的背景顏色與主體相同。
✅ 正確：深色主題的頁尾，明確將法律連結與頁面內容分開。
```

---

### 一致連接原則 (Law of Uniform Connectedness)

**原則：** 視覺上相連（例如：通過線條、箭頭）的元素比沒有連接的元素被感知為更相關。

**應用：**
- **流程：** 在進度嚮導中使用線條連接步驟。
- **選單：** 下拉選單應「觸碰」或連接到其父按鈕。
- **圖表：** 在圖表中連接數據點的線條。
- **關係：** 將切換開關連接到它所控制的文字。
- **層級：** 用於檔案目錄的樹狀結構。
- **表單：** 將「信用卡」單選按鈕連接到其下方的欄位組。

**範例：**
```
❌ 錯誤：1、2、3 步驟設置中數字分散。
✅ 正確：水平線連接 1、2、3 以顯示序列。

❌ 錯誤：漂浮的下拉選單沒有觸碰到開啟它的按鈕。
✅ 正確：下拉選單視覺上「附加」到父按鈕。
```

---

### 簡潔原則 (Law of Prägnanz / Simplicity)

**原則：** 人們會將模糊或複雜的圖像解釋為儘可能簡單的形式，因為這種解釋所需的認知負荷最少。

**應用：**
- **清晰度：** 使用清晰、幾何化的圖示進行導航。
- **縮減：** 移除不必要的 3D 紋理或陰影。
- **形狀：** 優先選擇標準的長方形/圓形而非複雜的多邊形。
- **焦點：** 為主要動作使用高對比度的剪影。
- **標誌：** 在小尺寸下也能識別的簡單品牌標誌。
- **UX：** 每頁一個主要目標，使「心理形狀」保持簡單。

**範例：**
```
❌ 錯誤：使用超寫實的 3D 資料夾插圖作為「檔案」圖示。
✅ 正確：簡單的 2D 資料夾輪廓。

❌ 錯誤：使用多色、複雜的標誌作為載入圖示。
✅ 正確：簡單的單色圓環。
```

---

### 主體/背景原則 (Law of Figure/Ground)

**原則：** 眼睛會將物件從周圍區域區分出來。形狀、剪影或輪廓被感知為主體（物件），而周圍區域被感知為背景。

**應用：**
- **焦點：** 使用遮罩 (Scrims) 讓彈窗內容突出。
- **深度：** 使用投影暗示「主體」位於「背景」之上。
- **對比：** 深色背景上的淺色文字（或反之）。
- **模糊：** 使用背景模糊來強調前景文字。
- **導航：** 漂浮的固定標頭，保持在頁面內容之上。
- **懸停：** 懸停時略微提升卡片，將其定義為主體。

**範例：**
```
❌ 錯誤：一個沒有陰影或邊框的彈窗，與頁面融為一體。
✅ 正確：帶有投影和變暗背景遮罩的彈窗。

❌ 錯誤：直接在雜亂的多色照片上放置白色文字。
✅ 正確：將白色文字放在半透明的深色「遮罩」上。
```

---

### 焦點原則 (Law of Focal Point)

**原則：** 任何在視覺上突出的東西都會首先捕捉並保持觀看者的注意力。

**應用：**
- **進入點：** 將主要價值主張放在焦點位置。
- **顏色：** 在中性 UI 中使用一個高彩度的「動作顏色」。
- **動效：** 在 CTA 上使用細微動效吸引目光。
- **尺寸：** 最重要的統計數據應使用最大的字體。
- **排版：** 標頭使用粗體，正文使用標準粗細。
- **方向：** 使用箭頭或視線引導（人看著按鈕的照片）。

**範例：**
```
❌ 錯誤：首頁有 5 個大小和顏色都相同的按鈕。
✅ 正確：一個色彩鮮艷的大型「開始使用」按鈕。

❌ 錯誤：儀表板上「總營收」的大小與「系統版本」相同。
✅ 正確：「總營收」顯示在頂部中央，字體巨大且粗體。
```

## 3. 認知偏誤與行為

### 蔡格尼效應 (Zeigarnik Effect)

**原則：** 人們對未完成或被中斷任務的記憶力優於已完成任務。

**應用：**
- **遊戲化：** 使用「個人資料已完成 60%」进度條。
- **參與度：** 在學習路徑中預告下一個模組。
- **留存：** 顯示一個尚未探索的功能「待辦清單」。
- **回饋：** 為未讀訊息顯示持久的徽章。
- **動量：** 完成一個步驟後立即顯示「下一步」。
- **購物：** 購物車中的「完成訂單」提醒。

**範例：**
```
❌ 錯誤：沉默的引導過程，沒有任何剩餘進度指示。
✅ 正確：顯示「5 步中的第 3 步已完成」的檢查清單。

❌ 錯誤：即便影片只看了一半，線上學習 App 也顯示打勾。
✅ 正確：進度環保持半滿，直到影片看完。
```

### 目標漸進效應 (Goal Gradient Effect)

**原則：** 隨著靠近目標，接近目標的傾向會增加。

**應用：**
- **動量：** 給予使用者「人為進展」（例如：2 個免費印章）。
- **進度：** 將 10 個欄位的表單拆分為兩個 5 欄位的步驟。
- **回饋：** 在任務完成一半時慶祝里程碑。
- **動機：** 向使用者展示他們距離獎勵/狀態有多近。
- **導航：** 使用導覽列 (Breadcrumbs) 顯示距離結束還有多近。
- **載入：** 當接近 100% 時加快載入動畫。

**範例：**
```
❌ 錯誤：從 0% 開始的進度條，讓人覺得路途遙遠。
✅ 正確：從 20% 開始的進度條，因為使用者打開 App 就算「開始」了。

❌ 錯誤：結帳流程中的「最終確認」感覺像是意外的第 5 步。
✅ 正確：明確標註步驟：「運送 > 付款 > 快完成了！」
```

### 峰終定律 (Peak-End Rule)

**原則：** 人們對體驗的評價很大程度上取決於其高峰期（最激烈的點）和結束時的感受，而不是每一刻感受的總和或平均值。

**應用：**
- **成功：** 讓「訂單確認」畫面令人難忘。
- **愉悅：** 在產生價值的點添加紙屑或獨特動畫。
- **支援：** 確保與聊天機器人的最後一次互動是有幫助的。
- **登出：** 即使使用者離開，也要讓最後的退場簡潔。
- **引導：** 以明確的「勝利」結束第一次工作會話。
- **錯誤處理：** 將 404 頁面變成有趣、有幫助的互動。

**範例：**
```
❌ 錯誤：20 分鐘的報稅過程後，App 只顯示「已提交」。
✅ 正確：一個帶有退稅金額摘要的「恭喜！」畫面。

❌ 錯誤：遊戲以簡單的普通字體文字「Game Over」結束。
✅ 正確：顯示高分摘要畫面，並配上慶祝音樂。
```

### 審美實用效應 (Aesthetic-Usability Effect)

**原則：** 使用者通常將美觀的設計視為更好用的設計。

**應用：**
- **信任：** 高保真視覺效果能為微小 Bug 贏得「信任額度」。
- **品牌：** 一致的高品質影像建立專業感。
- **參與度：** 美麗的介面讓使用者探索得更久。
- **耐心：** 如果 UI 漂亮，使用者對載入時間的容忍度更高。
- **信心：** 簡潔的設計讓複雜工具感覺更容易上手。
- **忠誠度：** 人們會與美麗的產品建立情感連結。

**範例：**
```
❌ 錯誤：一個文字對齊錯誤、配色土氣的銀行 App。
✅ 正確：一個擁有流暢動畫、時尚現代的銀行 App。

❌ 錯誤：使用低解析度、有像素感的庫存照片。
✅ 正確：使用高畫質、自定義的品牌插圖。
```

### 錨定偏誤 (Anchoring Bias)

**原則：** 使用者在做決定時過度依賴第一眼看到的資訊（「錨點」）。

**應用：**
- **定價：** 顯示被劃掉的原價。
- **層級：** 將最貴的「企業」方案放在最左邊。
- **排序：** 將「最熱門」作為第一個推薦高亮顯示。
- **折扣：** 在顯示最終價格前先說明「節省 20%」。
- **限制：** 「每位客戶限購 12 件」錨定了其高價值的印象。
- **預設值：** 從較高的「建議捐款」金額開始。

**範例：**
```
❌ 錯誤：僅顯示價格 "$49"。
✅ 正確：顯示 "~~$99~~ $49 (50% 折扣)"。

❌ 錯誤：將筆電列表從最便宜排到最貴。
✅ 正確：先顯示高階「Pro」型號，讓其他型號顯得便宜。
```

### 社會證明 (Social Proof)

**原則：** 人們會模仿他人的行為，藉此在給定情況下採取正確的行為。

**應用：**
- **驗證：** 顯示「加入其他 50,000+ 使用者」。
- **評論：** 星級評分和經過驗證的客戶見證。
- **標誌：** 顯示合作夥伴品牌的「信賴於」區域。
- **即時動態：** 「Sarah 5 分鐘前剛購買了這個」的通知。
- **活動：** 「目前有 300 人正在瀏覽此項目」。
- **證書：** 行業獎項和安全徽章。

**範例：**
```
❌ 錯誤：註冊頁面只有一個表單。
✅ 正確：註冊頁面寫著「加入 200 萬名設計師」。

❌ 錯誤：沒有名字或照片的匿名評論。
✅ 正確：包含頭像、姓名和「認證買家」標籤的評論。
```

### 稀缺原則 (Scarcity Principle)

**原則：** 人類對稀缺物件賦予更高價值，而對豐裕物件賦予較低價值。

**應用：**
- **緊迫感：** 「庫存僅剩 2 件」。
- **時間：** 促銷活動倒數計時器。
- **存取權：** 「僅限邀請」的測試版或專屬層級。
- **季節性：** 「夏季限定」產品。
- **低庫存：** 「即將補貨 — 立即預訂」。
- **需求：** 「需求量大 — 有 10 人已將此放入購物車」。

**範例：**
```
❌ 錯誤：一場永不結束且沒有倒數的促銷。
✅ 正確：帶有倒數計時器的「今日特惠」。

❌ 錯誤：顯示產品有貨，但沒有庫存數量。
✅ 正確：「此價格僅剩 3 件！」
```

### 權威偏誤 (Authority Bias)

**原則：** 傾向於將更高的準確性歸功於權威人士的意見，並更容易受其影響。

**應用：**
- **專業知識：** 使用「專家認證」或專業頭像。
- **認證：** 信任標章 (Norton, ISO, HIPAA)。
- **媒體：** 「曾見於 TechCrunch/Forbes」的標誌。
- **代言：** 行业領袖或網紅的見證。
- **語言：** 自信、專業且準確的文案。
- **歷史：** 「創立於 1950 年」暗指歷史悠久且值得信賴。

**範例：**
```
❌ 錯誤：由 "Admin" 撰寫的健康部落格。
✅ 正確：由「心臟科醫生 Jane Smith 博士審查」的健康文章。

❌ 錯誤：安全 App 沒有提到任何認證。
✅ 正確：顯示 "ISO 27001 Certified" 和 "Norton Secured" 標誌。
```

### 損失規避 (Loss Aversion)

**原則：** 人們通常比起獲得同等收益，更傾向於避免損失。不丟掉 5 美元比撿到 5 美元更好。

**應用：**
- **訊息：** 「不要錯過你的折扣」。
- **試用：** 「你的免費試用即將結束 — 立即保留你的資料」。
- **稀缺：** 「一旦錯過，就永遠消失了」。
- **購物車：** 「不要錯過購物車中的項目」。
- **忠誠度：** 「你已獲得 500 點 — 不要讓它們過期」。
- **風險：** 「30 天退款保證」（減少金錢的「損失」感）。

**範例：**
```
❌ 錯誤： 「點擊這裡獲得 10 美元優惠券」。
✅ 正確：「你有 10 美元抵用金等待領取。在今晚過期前使用它！」

❌ 錯誤： 「取消訂閱」。
✅ 正確：「如果你取消，你將失去 50 個已儲存專案的存取權」。
```

### 虛假共識效應 (False-Consensus Effect)

**原則：** 人們傾向於高估自己的意見、信念、偏好、價值觀和習慣在他人間的普遍程度。

**應用：**
- **測試：** 你不是使用者 — 請針對真實的受眾進行測試。
- **研究：** 使用定性數據（訪談）和定量數據（分析）。
- **偏見：** 使用「盲法設計評審」來避免個人偏好。
- **人物角色：** 堅持使用已建立的使用者人物角色 (Personas)，而非個人直覺。
- **變異：** 測試來自不同背景/能力的用戶。
- **客觀性：** 使用熱圖 (Heatmaps) 查看實際的使用者行為。

**範例：**
```
❌ 錯誤：設計師在沒有測試的情況下判定某項功能是「直觀的」。
✅ 正確：運行 A/B 測試來查看使用者偏好哪個版本。

❌ 錯誤：完全用英文開發 App，因為「每個人都懂英文」。
✅ 正確：根據實際的使用者位置數據添加本地化。
```

### 知識的詛咒 (Curse of Knowledge)

**原則：** 一種認知偏誤，發生在一個人在與他人溝通時，在不知不覺中假設對方擁有理解該主題所需的背景知識。

**應用：**
- **文案：** 避免術語，使用通俗語言。
- **引導：** 假設使用者一無所知的教程。
- **工具提示：** 懸停時解釋複雜術語。
- **結構：** 漸進式揭露（隱藏進階設定）。
- **標籤：** 為導航使用圖示 + 文字標籤（不要僅依賴圖示）。
- **支援：** 為初次使用者提供全面的常見問題解答 (FAQ)。

**範例：**
```
❌ 錯誤：錯誤訊息顯示 "Exception: Null Pointer at 0x0045"。
✅ 正確：錯誤訊息顯示「出錯了。請嘗試重新整理」。

❌ 錯誤：在雲端 App 中使用「S3 Bucket Instances」等術語進行導航。
✅ 正確：使用「檔案儲存」等簡單術語。
```

### 墊腳石效應 (Stepping Stone Effect / 登門檻效應)

**原則：** 如果使用者從小型任務開始，他們就會承諾完成大型任務。

**應用：**
- **漏斗：** 在要求信用卡之前先要求信箱。
- **參與度：** 在註冊前先詢問一個偏好（例如：「深色模式？」）。
- **引導：** 使用一系列「快速是非題」。
- **信任：** 在要求訂閱前先提供免費的 PDF/工具。
- **個人資料：** 先要求上傳照片，稍後再填寫自傳。
- **銷售：** 在主服務前提供低價的「試水溫」產品。

**範例：**
```
❌ 錯誤：一個立即需要信用卡資訊的「開始免費試用」按鈕。
✅ 正確：先要求信箱和密碼，然後再提供試用。

❌ 錯誤：一份在一個頁面上顯示所有 50 個問題的問卷。
✅ 正確：從一個簡單的「是非題」開始的問卷。
```

├─────────────────────────────────────────────────────────────┤
│  行為層 (Behavioral - 功能腦)                                  │
│  ─────────────────────────────                              │
│  • 易用性與功能                                              │

│  ─────────────────────────────                              │

## 3. 信任建立系統

### 信任信號類別

| 類別 | 元素 | 實作方式 |
|------|------|----------|
| **安全性** | SSL、徽章、加密 | 可見的掛鎖、表單上的安全標誌 |
| **社會證明** | 評論、見證、標誌 | 星級評分、客戶照片、品牌標誌 |
| **透明度** | 政策、定價、聯絡方式 | 清晰的連結、無隱藏費用、真實地址 |
| **專業度** | 設計品質、一致性 | 無損壞元素、一致的品牌形象 |
| **權威性** | 認證、獎項、媒體 | 「曾見於...」、行業認證 |

### 信任信號放置

```
┌────────────────────────────────────────────────────┐
│  標頭：信任橫幅 ("免運費 | 30 天退貨 | 安全結帳")   │
├────────────────────────────────────────────────────┤
│  HERO：社會證明 ("獲得 10,000+ 信任")             │
├────────────────────────────────────────────────────┤
│  產品：評論可見、安全徽章                         │
├────────────────────────────────────────────────────┤
│  結帳：支付圖示、SSL 徽章、保證                   │
├────────────────────────────────────────────────────┤
│  頁尾：聯絡資訊、政策、認證                       │
└────────────────────────────────────────────────────┘
```

## 4. 認知負荷管理

### 三種認知負荷

| 類型 | 定義 | 設計師的角色 |
|------|------|--------------|
| **內在 (Intrinsic)** | 任務固有的複雜性 | 拆分為較小步驟 |
| **外在 (Extraneous)** | 來自不良設計的負荷 | 消除這個！ |
| **關聯 (Germane)** | 用於學習的努力 | 支持並鼓勵 |

### 減輕策略

**1. 簡化 (減少外在負荷)**
```css
/* 視覺噪音 → 簡潔 */
.card-busy {
  border: 2px solid red;
  background: linear-gradient(...);
  box-shadow: 0 0 20px ...;
  /* 過多特效！ */
}

.card-clean {
  background: white;
  border-radius: 16px;
  box-shadow: 0 10px 30px -10px rgba(0,0,0,0.1);
  /* 冷靜、集中 */
}
```

**2. 資訊區塊化**
```html
<!-- 令人崩潰 -->
<form>
  <!-- 一次顯示 15 個欄位 -->
</form>

<!-- 區塊化 -->
<form>
  <fieldset>
    <legend>步驟 1：個人資訊</legend>
    <!-- 3-4 個欄位 -->
  </fieldset>
  <fieldset>
    <legend>步驟 2：運送資訊</legend>
    <!-- 3-4 個欄位 -->
  </fieldset>
</form>
```

**3. 漸進式揭露**
```html
<!-- 直到需要時才顯示複雜性 -->
<div class="filters">
  <div class="filters-basic">
    <!-- 常見篩選器可見 -->
  </div>
  <button onclick="toggleAdvanced()">
    進階選項 ▼
  </button>
  <div class="filters-advanced" hidden>
    <!-- 隱藏複雜篩選器 -->
  </div>
</div>
```

**4. 使用熟悉的模式**
```
✅ 標準導航位置
✅ 預期的圖示含義 (🔍 = 搜尋)
✅ 常規表單佈局
✅ 常見手勢模式 (滑動、捏合)
```

**5. 卸載資訊**
```html
<!-- 不要讓使用者死記硬背 -->
<label>
  卡號
  <input type="text" inputmode="numeric" 
         autocomplete="cc-number" 
         placeholder="1234 5678 9012 3456">
</label>

<!-- 顯示他們輸入過的內容 -->
<div class="order-summary">
  <p>運送至：<strong>John Doe, 123 Main St...</strong></p>
  <a href="#">編輯</a>
</div>
```

## 5. 說服式設計 (倫理)

### 倫理說服技術

| 技術 | 倫理使用 | 黑暗模式 (避免) |
|------|----------|-----------------|
| **稀缺性** | 真實庫存水平 | 虛假倒數計時器 |
| **社會證明** | 真實評論 | 虛假見證 |
| **權威性** | 真實憑證 | 誤導性徽章 |
| **緊迫感** | 真實截止日期 | 人造 FOMO |
| **承諾** | 保存進度 | 情緒勒索 |

### 助推 (Nudge) 模式

## 8. 心理學檢查清單

### 上線前

- [ ] **希克定律：** 導航中的選擇不超過 7 個。是否減少了選項以降低決策疲勞？
- [ ] **費茲定律：** 主要 CTA 夠大且易於觸及。最重要的按鈕在行動裝置上是否容易點擊？
- [ ] **米勒定律：** 內容已適當區塊化。資訊是否分組成 5-7 個易於消化的單元？
- [ ] **雅各布定律：** 網站是否遵循使用者已理解的標準 Web 常規？
- [ ] **多爾蒂門檻：** 系統是否在 400ms 內提供回饋？骨架屏是否已就位？
- [ ] **泰斯勒定律：** 複雜性是否已儘可能從使用者轉移到系統？
- [ ] **帕金森定律：** 是否有「一鍵結帳」等功能來縮短任務完成時間？
- [ ] **馮·雷斯托夫：** 主要 CTA 是否在視覺上與所有其他元素區分開來？
- [ ] **序列位置：** 最關鍵的資訊是否放在列表的最開始或最末尾？
- [ ] **格式塔定律：** 相關項目是否物理上分組在一起（接近原則）或放在卡片內（共同區域原則）？
- [ ] **蔡格尼效應：** 對於未完成的任務是否有視覺指示器（如進度條）？
- [ ] **目標漸進：** 是否給予使用者「領跑優勢」（例如：20% 進度）來鼓勵完成？
- [ ] **峰終定律：** 最終的「成功」畫面是否創造了驚喜時刻？
- [ ] **歐卡姆剃刀：** 是否移除了不必要的視覺或功能元素？
- [ ] **審美實用性：** UI 是否具有足夠的高保真度來建立最初的使用者信任？
- [ ] **信任與權威：** 安全徽章、評論和專家認證是否可見？
- [ ] **社會證明：** 真實的使用者數據或見證在決策點是否可見？
- [ ] **稀缺性與緊迫感：** 如果使用，稀缺性是否真實且符合倫理（例如：實際低庫存）？
- [ ] **損失規避：** 文案是否強調使用者能「保留」什麼，而不僅僅是獲得什麼？
- [ ] **錨定：** 價格呈現方式是否將所需的選擇框架為物超所值？
- [ ] **波斯塔爾定律：** 系統是否足夠靈活，能接受各種輸入格式而不報錯？
- [ ] **虛假共識：** 設計是否經過真實使用者的測試，而非僅由內部團隊決定？
- [ ] **知識的詛咒：** 文案是否沒有技術術語，且讓初學者易於理解？
- [ ] **墊腳石：** 漏斗是否從低摩擦任務開始（例如：僅輸入信箱）？
- [ ] **認知負荷：** 是否將外在的視覺噪音降至最低，以保持介面簡潔？
- [ ] **情感設計：** 色彩配置和影像是否喚起了預期的本能反應？
- [ ] **回饋：** 所有互動元素是否具有即時的懸停、點選和成功狀態？
- [ ] **無障礙：** 對比度是否足夠，網站是否可通過鍵盤/螢幕閱讀器導覽？
- [ ] **簡潔：** 圖示和形狀是否足夠簡單，一眼就能辨認？
- [ ] **主體/背景：** 焦點元素是否清晰（例如：為彈窗使用陰影或遮罩）？
