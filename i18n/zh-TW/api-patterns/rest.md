# REST 原則

> 基於資源的 API 設計 — 用名詞而非動詞。

## 資源命名規則

```
原則：
├── 使用名詞，而非動詞（資源，而非動作）
├── 使用複數形式（/users 而非 /user）
├── 使用小寫加連字號（/user-profiles）
├── 以巢狀表示關聯（/users/123/posts）
└── 保持淺層（最多 3 層深度）
```

## HTTP 方法選擇

| 方法 | 用途 | 冪等？ | 有主體？ |
|------|------|--------|----------|
| **GET** | 讀取資源 | 是 | 否 |
| **POST** | 建立新資源 | 否 | 是 |
| **PUT** | 替換整個資源 | 是 | 是 |
| **PATCH** | 部分更新 | 否 | 是 |
| **DELETE** | 移除資源 | 是 | 否 |

## 狀態碼選擇

| 情況 | 代碼 | 原因 |
|------|------|------|
| 成功（讀取）| 200 | 標準成功 |
| 已建立 | 201 | 新資源已建立 |
| 無內容 | 204 | 成功，無需回傳 |
| 錯誤請求 | 400 | 格式錯誤的請求 |
| 未授權 | 401 | 缺少/無效驗證 |
| 禁止 | 403 | 有效驗證，無權限 |
| 找不到 | 404 | 資源不存在 |
| 衝突 | 409 | 狀態衝突（重複）|
| 驗證錯誤 | 422 | 語法有效，資料無效 |
| 速率限制 | 429 | 請求過多 |
| 伺服器錯誤 | 500 | 我們的問題 |
